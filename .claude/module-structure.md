# Module Structure

- `cmd/appcli/` — `main` package; tiny shim that calls `cli.NewRootCommand().Execute()`. Cobra owns argv parsing, help, and shell-completion.
- `internal/cli/` — Cobra command tree. `root.go` builds the root command, persistent flags, and `PersistentPreRunE` (the flag-overlay step that completes the defaults → file → env → flags layering); `serve.go` is the daemon example using `app.RunUntilSignal`; `copy.go` is the one-shot example using `app.Run` and `app.Pool().SubmitWithContext` for recursive fan-out; `version.go` prints embedded build metadata and bypasses config loading. Add a new subcommand by writing a file here and registering it in `NewRootCommand`.
- `internal/application/` — `Application` type and CLI lifecycle (`Configure → Initialize → Run` / `RunUntilSignal`). The base `application.go` holds the struct, the lifecycle methods, the `Pool()` / `HealthRegistry()` accessors, and the `configSearchPaths` helper. Subsystem methods live in `application_<name>.go` files (`application_database.go`, `application_pool.go`, `application_health.go`); `signals_unix.go` / `signals_windows.go` provide the build-tagged shutdown-signal lists consumed by `RunUntilSignal`.
- `internal/pool/` — Goroutine-pool subsystem wrapping `github.com/panjf2000/ants/v2`. `config.go` carries `Size` / `NonBlocking` / `ExpiryDuration` / `PreAlloc` / `MaxBlockingTasks` (hydrated from `APP_POOL_*`); `pool.go` exposes `New`, `Submit`, `SubmitWithContext` (cancellation-aware), `Stats`, and `Release`/`ReleaseTimeout`. Submit errors come pre-wrapped in the `PoolErrors` errorx namespace.
- `internal/config/` — YAML/JSON config-file loader with multi-path search.
- `internal/env/` — Type-safe environment-variable parsers (string, bool, int, uint16, float, duration) and the `APP_` prefix convention.
- `internal/logging/` — One named zerolog logger (`Daemon`) plus the startup-notification helpers (`NotifyDaemonStartup`, `NotifyDaemonLoggingStartup`).
- `internal/errors/` — `joomcode/errorx` namespaces (`FileSystemErrors`, `DatabaseErrors`, `PoolErrors`); the canonical pattern for new error categories.
- `internal/health/` — Per-component health-check registry and the `Report` model. `health.go` defines `Status`, `ComponentResult`, `Report`, and `Registry` (Spring Boot Actuator-style aggregation: overall UP iff every component UP). The starter registers `lifecycle`, `pool`, and (when configured) `database` components from `Application.registerHealthChecks`; consumers can render the report from a future `health` subcommand or a periodic poller.
- `internal/database/` — Optional SQL database subsystem. `config.go` carries `Driver` + `DSN` (empty DSN ⇒ disabled and the rest is a no-op), `connection.go` wraps a mutex-guarded `*sql.DB` singleton with `Connect`/`Disconnect`/`Connection`/`IsHealthy`, `migration.go` runs Goose migrations embedded from `migrations/sql/*.sql`. Default driver is `pgx` (PostgreSQL).
- `internal/database/orm/` — Bun ORM singleton (`Configure`/`Database`/`Reset`). Add domain models as sibling files; the starter ships none.
- `internal/version/` — Build-time version metadata (commit, semver, tag) using `Masterminds/semver`. `commit.txt` is gitignored and populated by `go generate`.
- `internal/obfusicate/` — Small string-redaction helper exposing `ConcealPrefix` for masking values in log output. Spelling is intentional. Treat as out of scope unless the task explicitly references redaction.
- `docs/` — Non-Go documentation assets. `logo.svg` is embedded at the top of the README and is not consumed by any Go code.
- `.github/workflows/` — CI workflows, numbered per `.github/workflows/docs/naming-standards.md`. The 200/300-series flows run on PRs and `main` respectively; both delegate to 800-series reusables: `800-call-code-compiles.yaml`, `800-call-unit-test.yaml`, `800-call-vulncheck.yaml`. Releases run via `100-user-deploy-release-artifact.yaml` (manual `workflow_dispatch`) which delegates to `800-call-semantic-release.yaml`; that workflow handles the binary build, container OCI publish, and `actions/attest-*` Sigstore attestations for every artifact. Semantic-release plugin config lives in `.releaserc.json` at the repo root.
