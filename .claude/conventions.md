# Conventions

- **Functional options for module construction** — `module.WithRoutes(...)`, `module.WithSubModules(...)`, `module.WithMiddleware(...)`. Follow this pattern when adding new builder-style APIs.
- **No global state except loggers.** Configs and the `Application` instance flow through dependency wiring in `main.go`. Don't add package-level mutable state.
- **Echo v5, not v4** — import path is `github.com/labstack/echo/v5`. APIs differ from v4 in places (e.g. `StartConfig` for graceful shutdown, `*echo.Context` pointer handlers); verify against the v5 docs rather than assuming v4 patterns.
- **Errors are categorized via `joomcode/errorx` namespaces** (see `internal/errors/fs.go`). Don't introduce ad-hoc `errors.New` / `fmt.Errorf` for new error categories — define a namespace.
- **OpenAPI metadata via builder options, not comments.** Endpoints declare their request and response shapes inline via `endpoint.WithRequest(value)` / `endpoint.WithResponse(code, value, "description")` (and `WithSummary` / `WithDescription` for prose). The reflection-based schema generator at `internal/openapi/schema.go` converts the Go types into JSON Schemas at spec-build time. Don't introduce a comment-annotation grammar (swag-style) or a build-time AST generator — keeping metadata next to the handler in the route constructor matches the project's "everything is functional options" idiom and keeps schemas in the same source-of-truth as the routes that emit them.
- **Methods on `Application` go in `application_<subsystem>.go` files** (see `application_http.go`, `application_tls.go`, `application_proxy.go`, `application_database.go`, `application_health.go`, …). The base `application.go` holds only the struct definition and the lifecycle skeleton (`NewApplication`, `Configure`, `Initialize`, `RegisterModule`, `Start`, plus the `IsReady` accessor that's tied to the `ready` field). When you add a new subsystem to `Application`, create a new `application_<name>.go` with `configure<Name>`, `start<Name>` / `initialize<Name>`, and `shutdown<Name>` methods as appropriate, then call them from the lifecycle phases in `application.go`. Don't grow `application.go` itself.
